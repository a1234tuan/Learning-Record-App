# 编辑器增强设计 — 可行性分析

## 当前技术基础

编辑器基于 TipTap v2（ProseMirror 的 React 封装），已加载 StarterKit、TaskList、CodeBlockLowlight 等扩展。项目已有两个自定义节点——`RecordAssetNode`（资源卡片）和 `RecordFormulaNode`（KaTeX 公式）——均通过 `Node.create()` + `ReactNodeViewRenderer` 实现，即将一个 React 组件嵌入编辑器中作为一个不可拆分的原子块。

CSS 方面使用 CSS 自定义属性体系（`--color-*`、`--radius-*` 等），编辑器样式集中在 `pages.css` 的 `.editor-shell`、`.rich-editor`、`.editor-toolbar` 等选择器中。

---

## 一、关系链块

### 实现思路

新增一个自定义 TipTap 原子节点，数据存储在节点的 JSON 属性中（节点列表、旁注、分支关系）。节点内部用一个 React 组件渲染：横向排列的圆角矩形卡片，卡片之间用 CSS 或简易 SVG 画箭头连线，每个卡片下方有旁注输入框。

### 交互映射

- 回车添加下一个节点 → React 组件内部的输入框监听键盘事件，在数组末尾追加新节点
- 点击分叉按钮 → 在对应节点下挂载子节点数组，另起一行渲染
- 拖拽调序 → 通过 HTML 拖拽 API 或简单的手动上移/下移按钮调整数组顺序
- 填写旁注 → 每个卡片下方独立的文本输入框，更新节点对象的 note 字段

### 难度：中等偏高

| 方面 | 评估 |
|------|------|
| TipTap 集成 | 完全复用现有的原子节点模式（`RecordFormulaNode` 的套路），不需要突破新技术 |
| React 交互 | 数组增删改查 + 键盘事件 + 简易拖拽，常规 React 能力 |
| 箭头渲染 | 线性链条用纯 CSS（`::after` 伪元素画箭头）即可；分支树需要 SVG 叠加，略微复杂但可控 |
| 移动端适配 | 原子节点在 TipTap 中自带选中高亮和只读渲染，不需要额外处理 |
| 预估代码量 | 新建一个文件，约 250～350 行 |

### 结论：可行。项目现有的自定义节点模式是直接模板，只需要把交互做细致。分支形态可以先做线性链，树形分叉作为二期迭代。

## 二、卡片墙

### 实现思路

同样是一个 TipTap 原子节点，但内部的 React 组件本质上是一个迷你画布。每张卡片是一个绝对定位的子元素，位置通过鼠标/触摸拖拽来改变。卡片之间的连线用 SVG 叠加层绘制。所有卡片数据和连线数据存储在节点的 JSON 属性中。

### 交互映射

- 双击空白位置建卡片 → 在画布区域的点击坐标处创建新卡片对象
- 拖动卡片改变位置 → 监听 `pointerdown`/`pointermove`/`pointerup`，更新卡片的 x/y 坐标
- 从卡片边缘拖出连线 → 在 SVG 层中创建一条贝塞尔曲线，起点和终点绑定到两张卡片的边缘
- 切换卡片颜色 → 点击卡片上的色块按钮，更新 color 字段

### 难度：高

| 方面 | 评估 |
|------|------|
| TipTap 集成 | 原子节点模式可用，和关系链类似 |
| React 交互 | HTML 拖拽 API 在移动端的表现不稳定，需要自行实现 pointer 事件体系 |
| 触摸冲突 | 卡片墙需要响应触摸拖拽（移动卡片），但 ProseMirror 编辑器本身也响应触摸（滚动、选区），两者在同一区域内会相互冲突。需要在卡片拖拽开始时阻止编辑器接管触摸事件 |
| SVG 连线 | 拖拽连线需要在 SVG 层中动态更新路径的坐标，性能没问题但坐标计算在窗口缩放时需要刷新 |
| 内容编辑 | 卡片内部如果直接用可编辑元素，会和 TipTap 的选区管理产生冲突。最安全的做法是点击卡片后弹出一个浮动编辑框，而不是在卡片内直接编辑 |
| 预估代码量 | 新建两个文件，约 450～550 行 |

### 结论：可行但摩擦大。核心风险是移动端的触摸事件冲突——画布拖拽和编辑器滚动/选区互相抢占。建议以独立面板（而非内嵌编辑器块）的形式实现，可以完全避开 TipTap 的触摸处理，交互更流畅。

## 三、大纲树

### 实现思路

有两种路径。路径甲：做成 TipTap 原子节点，编辑态是一个简化的文本输入区，渲染态是树形图组件。路径乙：完全脱离 TipTap，做成编辑器中独立的 UI 区域，不参与 ProseMirror 的文档模型。

两种路径的数据模型相同：一个嵌套的树结构，每个节点有文本和子节点数组。

### 交互映射

- 编辑模式：文本区域中每行是一个节点，Tab/Shift+Tab 改变缩进层级，空行或不同缩进决定了父子关系
- 渲染模式：将树结构递归渲染为可视化树形图，从上到下展开
- 切换模式：编辑区和树形图之间一键切换，通过按钮触发

### 难度：低

| 方面 | 评估 |
|------|------|
| TipTap 集成 | 路径甲复用现有原子节点模式；路径乙完全不需要 TipTap 知识 |
| React 交互 | 核心逻辑是一个解析函数（缩进文本 → 嵌套 JSON）+ 一个递归渲染组件（嵌套 JSON → 树形 HTML）。两个都是纯函数，不涉及复杂事件处理 |
| 树形图渲染 | 用纯 CSS 实现：每层左侧加竖线（`border-left`），兄弟节点之间用横线连接（`::before` 伪元素）。不需要 SVG |
| 无触摸冲突 | 编辑模式是标准表单元素，渲染模式是只读展示，都不和编辑器手势竞争 |
| 预估代码量 | 新建一个文件，约 150～200 行 |

### 结论：最易实现。核心逻辑简洁，不涉及任何手感敏感的手势交互，也不依赖第三方库。路径乙更安全——独立于 ProseMirror 之外，不会和现有编辑功能互相影响。

## 四、标注照片

### 实现思路

增强现有的图片查看和展示逻辑，在图片上方叠加一个透明的标注层。标注元素（文本框、矩形框、箭头）存储为数组，每个元素有类型、位置、尺寸、文字内容、颜色等属性。标注数据可以存在对应 `RecordAssetNode` 的属性中。

### 交互映射

- 进入标注模式 → 点击图片上的"标注"按钮，图片上方出现透明标注层
- 放置文本框 → 点击图片任意位置，弹出输入框，输入文字后确认，文本框悬浮在该位置
- 画矩形框 → 点击"矩形"按钮后在图片上拖拽，松开后矩形框固定在拖拽区域
- 画箭头 → 点击"箭头"按钮后从起点拖到终点，生成一条带箭头的线
- 退出标注模式 → 标注元素保留在图片上方，整体作为一个视图展示

### 难度：中等

| 方面 | 评估 |
|------|------|
| TipTap 集成 | 不需要动 TipTap，改动集中在 `AssetPreview` 组件中——在图片容器内加一个标注覆盖层 |
| 坐标系统 | 需要处理标注坐标与图片实际尺寸的映射关系。如果图片在不同屏幕宽度下缩放，标注坐标需要基于百分比存储而非像素 |
| 拖拽绘制 | 矩形框和箭头的拖拽绘制是常规的 pointer 事件处理，需要区分"短按放置文本框"和"拖拽画矩形/箭头"两种意图 |
| 触摸冲突 | 标注模式下的拖拽和页面滚动需要互斥——进入标注模式后，标注层接管触摸事件，阻止页面滚动 |
| 预估代码量 | 修改 `AssetPreview.tsx` + 新建 `ImageAnnotation.tsx`，约 250～350 行 |

### 结论：可行。技术栈不涉及 TipTap 内部机制，改动集中在已有的图片组件上。唯一需要仔细处理的是坐标归一化——确保标注在不同屏幕宽度和对齐方式下不乱位。

## 五、折叠块

### 实现思路

在 TipTap 中创建一个可以包含子内容的容器节点，节点的标题行始终可见，内容区域可以折叠/展开。这要求使用 TipTap 的 `contentDOM` 模式——节点不是不可拆分的原子，而是包含一个可编辑的内容区，用户可以在这个区里继续使用加粗、列表、图片等所有正常编辑功能。

### 交互映射

- 插入折叠块 → 工具栏按钮新建一个折叠节点，默认状态为展开，标题行为占位文字
- 编辑标题 → 点击标题行直接编辑，纯文本输入
- 折叠/展开 → 点击标题行左侧的箭头图标或整行，切换内容区域的显示
- 编辑内容 → 内容区域是完整的富文本编辑区，所有工具栏功能可用
- 设置摘要 → 点击摘要占位区域输入一句话，在折叠状态下显示

### 难度：中等

| 方面 | 评估 |
|------|------|
| TipTap 集成 | 这是六个方向中唯一需要 `contentDOM` 模式的。项目现有的两个自定义节点都是原子节点（没有可编辑子内容），折叠块需要升级到 `contentDOM` 模式 |
| ReactNodeViewRenderer + contentDOM | TipTap 的 `ReactNodeViewRenderer` 支持 `contentDOM`——组件可以渲染标题栏等装饰 UI，同时暴露出一个 DOM ref 让 ProseMirror 管理其内部的可编辑内容。这是 TipTap 文档中已覆盖的常见模式 |
| 挑战 | 确保 ProseMirror 对内容区的更新（例如用户在其中输入文字）不会触发 React 重新渲染导致光标跳动。需要通过 `contentDOMElementRef` 正确地让 ProseMirror 接管这部分 DOM |
| 内容序列化 | 折叠块的内容是完整的 ProseMirror 文档片段，可以包含任意嵌套节点——这由 ProseMirror 自动处理，不需要手动序列化 |
| 预估代码量 | 新建 `CollapsibleBlockNode.tsx`，约 180～250 行 |

### 结论：可行。`contentDOM` + `ReactNodeViewRenderer` 是 TipTap 的标准模式，文档中有清晰的示例。风险比原子节点略高但可控——核心在于正确配置 contentDOM 的 ref 传递，避免 React 和 ProseMirror 互抢 DOM 控制权。

## 六、水平排列容器

### 实现思路

让多个内容块横向并排。这在 ProseMirror 中存在根本性的架构限制：ProseMirror 只有一个光标、一个选区，不能同时管理多个独立的可编辑列。可行的替代方案有两种：

方案甲：CSS 分栏——容器节点只有一个 `contentDOM`，通过 CSS `column-count` 让内容自动在视觉上分列。内容从第一列顶部流到底部，然后自动流入第二列。这是纯视觉效果，不改变编辑模型。

方案乙：原子列——容器内的每一列是一个独立的原子节点，内容以只读方式渲染，用户点击某列后弹出一个浮动编辑器在该列的内容中编辑，完成后再渲染回只读视图。

### 交互映射

方案甲中，用户像正常一样编辑，内容自动多列排列。方案乙中，用户看到的是只读渲染的并排卡片，点击某个卡片出现弹出式编辑框，编辑完成后卡片内容更新。

### 难度：高

| 方面 | 评估 |
|------|------|
| ProseMirror 架构限制 | ProseMirror 的单光标模型是这里无法绕过的硬约束。多列独立编辑在其设计哲学中不成立 |
| 方案甲 | CSS `column-count` 可以实现视觉分栏，但用户无法控制"哪些内容在哪一列"——内容是自然流动的。适合简单的文字对比，不适合需要精确控制每列内容的场景 |
| 方案乙 | 在技术上是可行的，但用户体验打了折扣——不能在行内编辑，需要弹出框。和"快速记录"的原则有张力 |
| 预估代码量 | 方案甲约 100 行（主要是 CSS + 一个简单容器节点）；方案乙约 300～400 行 |

### 结论：技术上可以实现（方案甲），但产品效果限制较大——分栏是纯视觉的，用户不能独立控制每列的内容边界。方案乙牺牲了编辑的自然感。整体来看这个方向的性价比不如其他五个方向。

## 总结对比

| 方向 | 技术路径 | 难度 | 风险点 | 可行 verdict |
|------|---------|------|--------|-------------|
| 关系链块 | 原子节点 + React 组件 | 中等偏高 | 分支树形的箭头布局 | ✅ 直接做 |
| 卡片墙 | 原子节点 + 迷你画布 | 高 | 触摸拖拽和编辑器滚动冲突 | ⚠️ 建议做独立面板 |
| 大纲树 | 独立 UI 或原子节点 | 低 | 无 | ✅ 直接做 |
| 标注照片 | 增强 AssetPreview | 中等 | 坐标归一化 | ✅ 直接做 |
| 折叠块 | contentDOM 节点 | 中等 | React/ProseMirror DOM 协作 | ✅ 直接做 |
| 水平容器 | CSS 分栏或原子列 | 高 | ProseMirror 单光标限制 | ⚠️ 降级为 CSS 分栏 |

**按实现顺序建议**：大纲树 → 折叠块 → 关系链块 → 标注照片 → 卡片墙 → 水平容器。前三个方向各覆盖不同的笔记场景（层级结构、主动回忆、线性流程），技术风险依次递增但都在可控范围。

**按对用户场景覆盖率排序**：关系链块 > 标注照片 > 大纲树 > 折叠块 > 卡片墙 > 水平容器。

如果只做三个，推荐组合是**关系链块 + 大纲树 + 折叠块**——三者互补，覆盖流程图（层次结构）、树形图（分类体系）、主动回忆（闭卷提取），且没有一个是高风险的。
